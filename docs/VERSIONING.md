# Versioning and Rollback

CRT는 기능이 바뀐 시점을 명확하게 남기고, 문제가 생겼을 때 안정된 이전 버전을 다시 사용할 수 있도록 버전별 기록을 유지합니다.

## Version Rules

| 표시 | 의미 | 예시 |
| --- | --- | --- |
| `0.x.0` | 베타 기능 업데이트 | `0.7.0` 포착 기록 저장 |
| `0.x.1` | 같은 기능 범위의 오류 수정 | `0.7.1` 저장 오류 수정 |
| `1.0.0` | 정식 배포 준비가 끝난 첫 출시 | 서명, 공증, 정책 검토 완료 후 결정 |

앱 화면의 간단한 표기는 `CRT 0.12`처럼 표시하고, Git 태그와 내부 버전은 `v0.12.0` 및 `0.12.0` 형식으로 남깁니다.

## Release Checklist

새 버전을 완료할 때마다 다음을 남깁니다.

1. 앱 표시 이름, 내부 버전, 배포 ZIP 파일명을 새 버전에 맞춥니다.
2. `CHANGELOG.md`에 추가 기능, 바뀐 동작, 현재 한계를 작성합니다.
3. README와 사용방법에서 실제 사용 흐름 및 데이터 범위를 갱신합니다.
4. 앱 빌드, 자동 테스트, 배포 ZIP 실행 검증을 수행합니다.
5. Git 커밋을 만들고 `v0.x.0` 태그를 해당 기능 커밋에 붙입니다.
6. GitHub로 커밋과 태그를 함께 Push합니다.

## Rollback Use

문제가 생긴 경우 가장 단순한 복구는 정상 작동했던 버전의 ZIP 앱을 다시 사용하는 것입니다. 코드를 다시 살펴보거나 새 복구 버전을 만들 때는 Git 태그를 기준점으로 사용합니다.

| 필요한 상황 | 처리 |
| --- | --- |
| 최신 앱만 문제가 있음 | 이전 버전 ZIP을 다시 실행 |
| 이전 코드를 확인해야 함 | 해당 버전 태그에서 코드를 열어 비교 |
| 새 기능은 유지하고 오류만 수정 | 다음 패치 버전으로 고쳐 배포 |
| 저장 데이터 형식 변경 후 문제가 있음 | 데이터 백업본을 보존한 상태에서 호환 복구 버전을 제작 |

## Stored Data From 0.7

`0.7`부터 포착 기록 데이터베이스가 추가되었습니다. 앱 버전만 되돌리는 것으로는 충분하지 않을 수 있습니다. 이전 앱이 새로운 데이터 구조를 읽지 못할 수 있기 때문입니다.

앞으로 저장 기능에는 다음 원칙을 적용합니다.

- 데이터베이스에 스키마 버전을 기록합니다.
- 데이터 구조를 변경하기 전에 기존 기록을 백업하거나 내보낼 수 있게 합니다.
- 앱을 낮은 버전으로 돌릴 때 원본 기록을 자동 삭제하거나 덮어쓰지 않습니다.
- 버전 기록에는 데이터 호환성 변화도 함께 적습니다.

`CRT 0.7.0`은 SQLite 저장소에 스키마 버전 `1`을 기록합니다. `CRT 0.8.0`은 2차 조사 보고 열을 추가한 포착 기록 스키마 버전 `2`로 올리며, `CRT 0.9.0`, `CRT 0.10.0`, `CRT 0.11.0`, `CRT 0.12.0`도 같은 포착 기록 구조를 유지해 기존 표본을 보존합니다. `CRT 0.10.0`은 별도의 `supporter-training.sqlite` 스키마 버전 `1`을 추가했고, `CRT 0.11.0`은 같은 저장소를 스키마 버전 `2`로 올려 고점·후속 성과·위험 라벨을 보존합니다. `CRT 0.12.0`은 스키마 버전 `3`으로 뉴스·SEC 근거 요약을 추가합니다. 기록 화면의 `CSV 내보내기`로 포착 표본을 별도 보관할 수 있습니다.
